GitHub रिपॉजिटरी के लिए "अपनाना" नियंत्रण प्रणाली। यह एक "फ़ॉर्क" रैपर नहीं है, बल्कि एक समग्र परत है जो "अपनाना" की तैयारी का आकलन करती है, सही डुप्लिकेशन पथ का चयन करती है, इसे एक ट्रैक किए गए एसिंक्रोनस ऑपरेशन के रूप में निष्पादित करती है, परिणाम को चलाने योग्य रखती है, समय के साथ इसे सिंक्रोनाइज़ करती है, और - v1.1.0 में नया - जब आप इसे अपना बनाने के लिए तैयार हों तो इसका नाम सुसंगत रूप से बदल देती है।

## v1.1.0 में नया क्या है

लेयर 7 - **AST-जागरूक बहुभाषी नाम बदलना।** `forkctl rename plan` पहचान फ़ाइलों, कोड प्रतीकों (26 भाषाएं ast-grep के माध्यम से) और गैर-कोड पाठ्य सतहों में एक समीक्षा योग्य अंतर उत्पन्न करता है। `forkctl rename apply` ट्री का स्नैपशॉट लेता है, सभी चरणों को चलाता है, लॉकफ़ाइलों को फिर से उत्पन्न करता है, और किसी भी बाइनरी फ़ाइल के लिए एक संपत्ति-पुनर्जनन मैनिफेस्ट छोड़ता है। `forkctl rename rollback` नवीनतम स्नैपशॉट को पुनर्स्थापित करता है। इसमें कोई `sed` श्रृंखला नहीं है। यह शब्द सीमाओं को सही ढंग से संभालता है। यह केस-संवेदनशील है।

## Forkctl क्या करता है

GitHub रिपॉजिटरी को "फ़ॉर्क" करना एक क्लिक का काम है। इसे "अपनाना" - यानी, "फ़ॉर्क" बनाम "टेम्प्लेट" चुनना, संगठन की नीतियों से निपटना, एसिंक्रोनस निर्माण की प्रतीक्षा करना, अपस्ट्रीम सिंक को कॉन्फ़िगर करना, और परिणाम को वास्तव में चलाने योग्य बनाना - बाकी सब कुछ है।

Forkctl बाकी सब कुछ का ध्यान रखता है।

| लेयर | यह क्या करता है |
|--------------|-----------------------------------------------------------------------------------------------|
| आकलन | किसी रिपॉजिटरी की एडॉप्शन की तैयारी का स्कोर करें, "फ़ॉर्क" बनाम "टेम्प्लेट" बनाम "इम्पोर्ट" की सिफारिश करें, स्रोत-पक्षीय सुधारों का सुझाव दें। |
| निष्पादन | कॉपी को एक ट्रैक किए गए एसिंक्रोनस ऑपरेशन के रूप में बनाएं। संगठन/एंटरप्राइज "फ़ॉर्क" नीतियों में आने वाली बाधाओं को शुरुआत में ही उजागर करता है। |
| बूटस्ट्रैप | प्रोफ़ाइल-आधारित पोस्ट-केयर - अपस्ट्रीम कॉन्फ़िगरेशन, README अपडेट, "ड्रिफ्ट" स्कैन, चलाने योग्य हैंडऑफ़। |
| सिंक | GitHub "मर्ज-अपस्ट्रीम" API को कॉल करता है। ईमानदारी से विचलन की रिपोर्ट करता है। आवश्यकता पड़ने पर "पुल रिक्वेस्ट" (PR) का उपयोग करता है। |
| फ़्लीट | अपने "फ़ॉर्क" की सूची बनाएं, उनकी स्थिति जांचें और उन्हें बैच में सिंक करें। |
| रसीदें | प्रत्येक ऑपरेशन का मशीन-पठनीय रिकॉर्ड। स्थानीय SQLite में ऑडिट लॉग। |
| नाम बदलना | AST-जागरूक बहुभाषी नाम बदलना - पहचान फ़ाइलें, कोड प्रतीक, पाठ्य सतहें, लॉकफ़ाइल पुनर्जनन। |

## उपयोग के तरीके

Forkctl को एक **MCP सर्वर** (स्टैंडर्ड इनपुट/आउटपुट ट्रांसपोर्ट, क्लाउड कोड और अन्य MCP क्लाइंट के लिए) और एक **CLI** दोनों के रूप में उपलब्ध कराया गया है, जिसमें समान कार्यक्षमता है।

### MCP

अपने MCP क्लाइंट कॉन्फ़िगरेशन में जोड़ें:

```json
{
  "mcpServers": {
    "forkctl": {
      "command": "npx",
      "args": ["-y", "@mcptoolshop/forkctl", "mcp"],
      "env": { "GITHUB_TOKEN": "ghp_..." }
    }
  }
}
```

### CLI

```bash
npx @mcptoolshop/forkctl assess owner/repo
npx @mcptoolshop/forkctl choose-path owner/repo --goal contribute_upstream
npx @mcptoolshop/forkctl create-fork owner/repo --destination-org my-org
npx @mcptoolshop/forkctl sync my-fork
npx @mcptoolshop/forkctl fleet-health
```

सभी कमांड मशीन-पठनीय आउटपुट के लिए `--json` स्वीकार करते हैं।

## बीस-दो उपकरण
<!-- FORKABLE_COUNTS_END -->

### आकलन
- `forkctl_assess` — एडॉप्शन की तैयारी का स्कोर, बाधाएं, ताकतें
- `forkctl_choose_path` — "फ़ॉर्क" | "टेम्प्लेट" | "इम्पोर्ट" | "क्लोन-डिटैच्ड"
- `forkctl_make_forkable` — स्रोत रिपॉजिटरी को ठीक करें (डिफ़ॉल्ट: योजना; वैकल्पिक: PR)

### निष्पादन
- `forkctl_preflight_policy` — एंटरप्राइज/संगठन/रिपॉजिटरी "फ़ॉर्क" नीतियों में आने वाली बाधाओं का पता लगाएं
- `forkctl_create_fork` — एसिंक्रोनस, ऑपरेशन आईडी लौटाता है
- `forkctl_create_from_template` — GitHub `/generate` का उपयोग करता है
- `forkctl_check_operation` — किसी भी चल रहे ऑपरेशन की जांच करें

### बूटस्ट्रैप
- `forkctl_bootstrap` — प्रोफ़ाइल-आधारित (योगदानकर्ता / स्टार्टर-किट / आंतरिक-सीड / क्लाइंट-डिलीवरी / प्रयोग)
- `forkctl_configure_upstream` — रिमोट सेट करें, वैकल्पिक सिंक वर्कफ़्लो
- `forkctl_scan_drift` — हार्डकोडेड पथ, लीक हुई सीक्रेट जानकारी, कॉपी में पुराने CI संदर्भ
- `forkctl_emit_handoff` — एकल, सटीक जानकारी: URL, कमांड, चेतावनी, अगला कदम

### सिंक
- `forkctl_sync` — GitHub "मर्ज-अपस्ट्रीम" API
- `forkctl_diagnose_divergence` — पीछे छूटे हुए कमिट, जोखिम वाले फ़ाइलें, संभावित टकराव
- `forkctl_propose_sync_pr` — "फ़ास्ट-फ़ॉरवर्ड" विफल होने पर PR-आधारित सिंक

### फ़्लीट
- `forkctl_list_forks` — आपके और देखे गए "फ़ॉर्क", स्वास्थ्य कॉलम के साथ
- `forkctl_fleet_health` — पुराने / टकराव वाले / परित्यक्त
- `forkctl_batch_sync` — सीमित, दर-सीमा-जागरूक

### रसीदें
- `forkctl_receipt` — किसी भी ऑपरेशन का मशीन-पठनीय रिकॉर्ड
- `forkctl_audit_log` — केवल अपेंड इतिहास

### नाम बदलना (लेयर 7 - v1.1.0 में नया)
- `forkctl_rename_plan` - AST-जागरूक नामकरण योजनाकार; समीक्षा योग्य अंतर उत्पन्न करता है।
- `forkctl_rename_apply` - स्नैपशॉट + पहचान, प्रतीक, पाठ्य और पोस्ट चरणों को लागू करता है।
- `forkctl_rename_rollback` - नवीनतम स्नैपशॉट से पुनर्स्थापित करता है।

## बूटस्ट्रैप प्रोफ़ाइल

| प्रोफ़ाइल | के लिए | पोस्ट-केयर |
|---------------------|--------------------------------------------------------------------|---------------------------------------------------------------------------------|
| `contributor`       | "फ़ॉर्क" करके अपस्ट्रीम को PR भेजना | अपस्ट्रीम रिमोट, सिंक वर्कफ़्लो, योगदानकर्ता README ब्लॉक, यदि अनुपस्थित हो तो PR टेम्पलेट |
| `starter-kit`       | एक टेम्पलेट से उत्पन्न, ताकि आप अपना उत्पाद शुरू कर सकें। | टेम्प्लेट संदर्भों को हटाएं, नया README फ़ाइल, नया LICENSE संदेश, .env.example फ़ाइल। |
| `internal-seed`     | एक साझा रिपॉजिटरी की आंतरिक टीम की प्रतिलिपि। | प्लेसहोल्डर्स को बदलें, आंतरिक CODEOWNERS सेट करें, दृश्यता को लॉक करें। |
| `client-delivery`   | प्रत्येक क्लाइंट के लिए एक अलग संस्करण (डिलीवरेबल)। | क्लाइंट के नाम से ब्रांच, इतिहास की जांच (सुरक्षा के लिए), डिफ़ॉल्ट ब्रांच को लॉक करना। |
| `experiment`        | अस्थायी/अलग कॉपी। | अपस्ट्रीम से अलग करें, README में "प्रयोग" के रूप में चिह्नित करें, कोई सिंक कार्यप्रवाह नहीं। |

## कॉन्फ़िगरेशन।

| चर। | आवश्यक। | डिफ़ॉल्ट। | टिप्पणियाँ। |
|----------------------|----------|----------------------------------------------|-------------------------------------------------|
| `GITHUB_TOKEN`       | हाँ। | —                                            | `repo`, `workflow`, `read:org` अनुमतियाँ। |
| `GITHUB_API_URL`     | no       | `https://api.github.com`                     | GHES / ghe.com के लिए। |
| `FORKCTL_STATE_DIR` | no       | ऑपरेटिंग सिस्टम उपयोगकर्ता-स्टेट निर्देशिका (env-paths के माध्यम से)। | जहां SQLite ऑपरेशन और ऑडिट डेटाबेस मौजूद हैं। |

## सुरक्षा।

खतरे के मॉडल और रिपोर्टिंग नीति के लिए [SECURITY.md](SECURITY.md) देखें। मुख्य बातें:

- `GITHUB_TOKEN` कभी भी लॉग नहीं किया जाता है।
- प्रत्येक टूल इनपुट को Zod के माध्यम से सत्यापित किया जाता है।
- `make_forkable` डिफ़ॉल्ट रूप से `plan` मोड में होता है। `pr` मोड वैकल्पिक है।
- Forkctl कभी भी फोर्स-पुश नहीं करता है, रिपॉजिटरी को नहीं हटाता है, या ब्रांच को नहीं हटाता है।
- कोई टेलीमेट्री नहीं। केवल कॉन्फ़िगर किए गए GitHub API को छोड़कर कोई भी आउटबाउंड कॉल नहीं।
